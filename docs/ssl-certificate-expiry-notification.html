Executive Summary

This solution is an Azure Automation Python 3 Runbook that proactively monitors SSL/TLS certificate expiry across all Azure Application Gateways in one or more subscriptions. It runs on a daily schedule, scans every certificate attached to a live TLS listener, and delivers a colour-coded health report — an HTML email plus a full Excel attachment — via Azure Communication Services (ACS) Email.

The runbook is entirely read-only: it never creates, modifies, or deletes any Azure resource.

Solution Overview

An Azure Automation Python 3 Runbook executes on a daily schedule to perform the full certificate health check automatically:

  1. Authenticate to Azure using a Service Principal (ClientSecretCredential)
  2. Discover every Application Gateway across all configured subscriptions
  3. Fetch full gateway details including publicCertData for each SSL certificate
  4. Parse certificate expiry dates from PKCS#7/DER/PEM encoded certificate data
  5. Classify each certificate: Expired / Critical / Warning / OK / Unknown
  6. Report results via a colour-coded HTML email and a full Excel attachment

Certificate Status Thresholds

StatusCondition
ExpiredDays remaining < 0 — certificate has already expired
Critical0 – 7 days remaining — requires immediate action
Warning8 – 14 days remaining (configurable via AG_ALERT_DAYS)
OK> 14 days remaining — certificate is healthy
UnknownExpiry could not be parsed (e.g. Key Vault-backed certificates)

Execution Flow

Detailed Phase Breakdown

PhaseActions
AuthenticateReads AzureSPCredential from Automation Account; builds ClientSecretCredential; instantiates NetworkManagementClient per subscription
Discover & ScanCalls list_all() to enumerate all Application Gateways; calls get() per gateway for full publicCertData; filters to certificates attached to active TLS listeners
Parse & ClassifyAttempts 4-stage fallback decode: PKCS#7 → bare DER → PEM-in-base64 → raw PEM; classifies each cert by days remaining; marks Unknown if expiry cannot be parsed
Build ReportsBuilds colour-coded Excel workbook sorted by days remaining; builds HTML email with metrics card and per-status top-5 tables
SendSends HTML email + Excel attachment via ACS Email; DRY_RUN=true writes to disk for local testing
4-stage certificate parsing: Azure Application Gateways return certificate data in multiple formats depending on source and version. The 4-stage fallback chain (PKCS#7 → DER → PEM-in-base64 → raw PEM) ensures maximum compatibility across all gateway configurations.

Key Technical Highlights

Resilient Certificate Parsing

  • Four-stage fallback decode chain handles every certificate format Azure may return
  • PKCS#7 container → bare DER → PEM-in-base64 → raw PEM string
  • Unknown status clearly flagged for Key Vault-backed certificates where publicCertData is not exposed

Listener-Aware Scanning

  • Only certificates attached to active TLS listeners are reported
  • Orphaned / unused certificates are intentionally skipped to reduce noise in reports

Read-Only Safety

  • Zero write operations against any Azure resource
  • Safe to run with minimal RBAC permissions — Reader role only
  • DRY_RUN=true mode writes output to disk without sending any email

Dual Output Format

  • HTML email — at-a-glance metrics card with colour-coded per-status tables
  • Excel attachment — full 13-column report with auto-filter and row-level colour coding

Configuration-Driven

  • All parameters stored in Automation Account variables — no redeployment needed to update config
  • Warning threshold (AG_ALERT_DAYS), subscription list, and recipient list are all runtime-configurable

IAM & Security Model

Application Gateway scanningService Principal (AzureSPCredential)Reader role on each target subscription
Email deliveryACS connection stringAuthenticated ACS Email sender permission

Required RBAC

The Service Principal requires Reader role on each target subscription (or at minimum on resource groups containing Application Gateways). No write permissions are required — the runbook is entirely read-only.

Security Guidelines

These rules apply to every deployment of this runbook:
  • Store all secrets in Azure Automation encrypted variables / credentials — never hardcode in the runbook source
  • The .env file is for local development only — add it to .gitignore and never commit to source control
  • Scope the Service Principal to the minimum required subscriptions with Reader role only
  • Rotate the Service Principal client secret on your organisation's standard credential rotation schedule

Operational Runbook

Trigger the Runbook Manually

Navigate to the Automation Account in the Azure Portal → Runbooks → select the runbook → Start.

Test Locally

pip install -r requirements.txt
DRY_RUN=true python main.py   # writes .xlsx + .html to disk, no email sent

Change the Warning Threshold

Update the AG_ALERT_DAYS variable in the Automation Account. No runbook redeployment is required.

Add a New Subscription

  1. Add the subscription ID to the AG_SUBSCRIPTION_IDS Automation variable (comma-separated)
  2. Grant the Service Principal Reader role on the new subscription

Add or Change Email Recipients

Update the AG_NOTIFICATION_TO Automation variable. No runbook redeployment is required.

Check Execution History

Navigate to the Automation Account → Jobs to view past executions, output logs, and error details.
