# Azure Application Gateway SSL Certificate Expiry Notification

## Executive Summary

This solution is an **Azure Automation Python 3 Runbook** that proactively monitors SSL/TLS certificate expiry across all Azure Application Gateways in one or more subscriptions. It runs on a schedule, scans every certificate attached to a live TLS listener, and delivers a colour-coded health report — an HTML email (top-5 most urgent per status) plus a full Excel attachment — via **Azure Communication Services (ACS) Email**.

The runbook is entirely **read-only**: it never creates, modifies, or deletes any Azure resource.

---

## What It Does

1. **Authenticates** to Azure using a Service Principal (`ClientSecretCredential`).
2. **Discovers** every Application Gateway across all configured subscriptions via `ApplicationGateways.list_all()`.
3. **Fetches** full gateway details per-gateway with `ApplicationGateways.get()` (the list API omits `publicCertData`).
4. **Parses** each SSL certificate's expiry date from the Azure `publicCertData` field, which is a base64-encoded PKCS#7 (`.p7b`) container. Four fallback decode strategies are attempted (PKCS#7 → bare DER → PEM-in-base64 → raw PEM string).
5. **Skips** certificates not attached to any HTTP listener (orphaned / unused certs).
6. **Classifies** each certificate:
   | Status | Condition |
   |--------|-----------|
   | Expired | days remaining < 0 |
   | Critical | 0 – 7 days remaining |
   | Warning | 8 – 14 days remaining (configurable) |
   | OK | > 14 days remaining |
   | Unknown | Expiry could not be parsed (e.g. Key Vault-backed certs) |
7. **Builds** a colour-coded Excel workbook (`ssl_certificate_report_<date>.xlsx`) with all certificates.
8. **Builds** a branded HTML email showing the top-5 most urgent certificates in each alert status, plus an "Unknown" table.
9. **Sends** both via Azure Communication Services (ACS) Email to a configurable recipient list.

## Components

### `main.py` — Core Runbook

The single-file runbook is divided into six sections:

| Section | Responsibility |
|---------|---------------|
| **Section 1 — Configuration** | Resolves all settings from Azure Automation variables/credentials, environment variables, or hard-coded defaults. `DRY_RUN=true` writes outputs to disk instead of sending email. |
| **Section 2 — Certificate Scanning** | Read-only Azure API calls. Lists all Application Gateways, GETs each for full cert data, parses PKCS#7/DER/PEM blobs, builds flat records. |
| **Section 3 — Excel Report** | Builds a colour-coded `.xlsx` workbook using `openpyxl`. Rows sorted by days remaining (soonest first). |
| **Section 4 — HTML Email** | Builds a branded, responsive HTML email with per-status tables (Expired / Critical / Warning / Unknown) and an at-a-glance metrics card. |
| **Section 5 — ACS Email** | Sends the HTML email + Excel attachment via `azure-communication-email`. |
| **Section 6 — Entry Point** | `main()` orchestrates sections 1–5. |

### `requirements.txt` — Dependencies

| Package | Purpose |
|---------|---------|
| `azure-identity` | Service Principal authentication |
| `azure-mgmt-network` | Application Gateway list/get API |
| `azure-mgmt-subscription` | Resolve subscription display names |
| `azure-communication-email` | Send email via ACS |
| `cryptography` | Parse PKCS#7 / X.509 certificate data |
| `openpyxl` | Build Excel `.xlsx` workbook |
| `python-dotenv` | Load `.env` for local runs (not installed in Automation Account) |

### `setup_automation_packages.py` — Package Uploader

One-time setup script that:
1. Uses `pip download` to resolve all transitive dependencies for Linux Python 3.8 (the Azure Automation runtime).
2. Reads wheel metadata and performs a topological sort (dependencies before dependants).
3. Looks up each wheel's public PyPI URL.
4. Uploads each wheel to the Azure Automation Account via `az automation python3-package create`.
5. Polls until each package reaches `Succeeded` state before proceeding to the next.

### `test_acs_email.py` — ACS Smoke Test

A standalone script that sends a simple plain-text + HTML email to verify the ACS connection string and sender domain are correctly configured. Useful for validating ACS credentials before deploying the full runbook.

| Resource | Purpose |

### Required RBAC Permissions

The Service Principal needs **Reader** role on each target subscription (or at least on the resource groups containing Application Gateways). No write permissions are required.

## Configuration

### Azure Automation Account Assets

Configure these in the Automation Account before running the runbook in production:

| Asset type | Name | Value |
|------------|------|-------|
| Credential | `AzureSPCredential` | Username = Client ID, Password = Client Secret |
| Variable (String) | `AZURE_TENANT_ID` | Azure AD tenant ID |
| Variable (String) | `AG_SUBSCRIPTION_IDS` | Comma-separated subscription IDs |
| Variable (String) | `AG_ALERT_DAYS` | Days before expiry for Warning threshold (default: `14`) |
| Variable (String) | `ACS_CONNECTION_STRING` | ACS Email resource connection string |
| Variable (String) | `ACS_SENDER_ADDRESS` | Verified ACS sender address |
| Variable (String) | `AG_NOTIFICATION_TO` | Comma-separated recipient email addresses |

### Local Development (`.env` file)

For local runs, create a `.env` file with the following keys (see `.env` as an example — **never commit real secrets**):

```dotenv
AZURE_CLIENT_ID=<service-principal-client-id>
AZURE_CLIENT_SECRET=<service-principal-client-secret>
AZURE_TENANT_ID=<tenant-id>
AG_SUBSCRIPTION_IDS=<sub-id-1>,<sub-id-2>
AG_ALERT_DAYS=14
DRY_RUN=false
ACS_CONNECTION_STRING=<acs-connection-string>
ACS_SENDER_ADDRESS=DoNotReply@<domain>.azurecomm.net
AG_NOTIFICATION_TO=team@example.com
```

Set `DRY_RUN=true` to skip the ACS call and write the Excel and HTML files to disk locally for inspection.

## Deployment Guide

### 1. Create the Automation Account and Upload Packages

```bash
# Edit RESOURCE_GROUP and AUTOMATION_ACCOUNT at the top of the script first
python setup_automation_packages.py
```

This only needs to be run once (or when `requirements.txt` changes).

### 2. Import the Runbook

Upload `main.py` as a **Python 3** runbook in the Automation Account.

### 3. Configure Assets

Add the Automation Account variables and credential listed in the Configuration section above.

### 4. Create a Schedule

Link a daily schedule to the runbook (e.g. 06:00 UTC).

### 5. Test Locally

```bash
pip install -r requirements.txt
DRY_RUN=true python main.py          # writes xlsx + html to disk, no email sent
python test_acs_email.py             # verifies ACS Email connectivity
```

## Email Output

### HTML Email

- **Header card:** "At a glance" metrics — total certs scanned, alert badge counts, report timestamp.
- **Expired table:** Top-5 expired certificates (dark red theme).
- **Critical table:** Top-5 certificates expiring within 7 days (orange-red theme).
- **Warning table:** Top-5 certificates expiring within 14 days (amber theme).
- **Unknown table:** All certificates whose expiry could not be determined.
- **Legend** and automated disclaimer footer.

### Excel Attachment

Full report for all certificates, sorted by days remaining (soonest first). Columns:

`Gateway Name` · `Location` · `Resource Group` · `Subscription Name` · `Subscription ID` · `Certificate Name` · `Listeners` · `Subject` · `Issuer` · `Expiry Date (UTC)` · `Days Remaining` · `Status` · `Key Vault Ref`

Each row is colour-coded: red (Expired), orange-red (Critical), orange (Warning), green (OK), grey (Unknown).

## Benefits

- **Zero-touch monitoring** — runs daily on a schedule, no manual effort required after initial setup.
- **Multi-subscription** — a single runbook covers all Application Gateways across an entire Azure estate.
- **Read-only** — no risk of accidental resource modification; safe to run with minimal RBAC permissions.
- **Dual output** — HTML email for at-a-glance visibility, Excel for audit records and detailed review.
- **Resilient parsing** — four-stage fallback chain handles the various certificate formats Azure may return.
- **Dry-run mode** — outputs to disk for local testing without any Azure Email dependency.
- **Status thresholds** — configurable warning window (`AG_ALERT_DAYS`); four clearly defined severity levels.
- **Listener-aware** — skips orphaned certificates not attached to any TLS listener, reducing noise.

---

## Limitations

- **Key Vault-backed certificates** — Azure does not expose `publicCertData` for certificates sourced from Key Vault. These appear as "Unknown expiry" in the report and must be monitored separately (e.g. via Azure Key Vault expiry alerts).
- **No auto-renewal** — the runbook is notification-only. It does not renew, replace, or rotate any certificate.
- **Azure Automation Python 3.8 runtime** — the runbook must remain compatible with Python 3.8 (the version available in Azure Automation at time of writing). Some newer library features may require version pinning.
- **ACS Email single-region** — ACS Email is region-specific; the connection string is tied to the ACS resource region.
- **No historical trend** — each run is independent. There is no built-in storage of past results or trending data.
- **Certificates without listeners skipped** — orphaned certificates not assigned to a listener are intentionally excluded; manual review is needed to identify them.
- **Network-level certificates not covered** — the runbook only scans Application Gateway SSL certificates. Certificates used on other Azure services (Front Door, API Management, App Service, etc.) are out of scope.

---

## Security Considerations

- Store all secrets (Client Secret, ACS connection string) exclusively in **Azure Automation encrypted variables / credentials** — never hard-code them in the runbook.
- The `.env` file is for local development only — add it to `.gitignore` and never commit it to source control.
- `test_acs_email.py` should not contain hard-coded credentials in production; read all values from environment variables or a secrets manager.
- The Service Principal should be scoped to the minimum required subscriptions with **Reader** role only.
